Overview of Storage Categories and Mediums

In forensic investigations, storage devices fall into three main categories—primary, secondary, and tertiary—based on how data persists and the role each plays. Beneath these categories are distinct technologies (magnetic, flash, optical, cloud) that dictate choice of tools and processes.

  • Primary Storage: Volatile memory like RAM and CPU cache. Holds running processes, encryption keys, and in-flight data that vanishes on power-off.
  • Secondary Storage: Non-volatile devices such as HDDs, SSDs, USB sticks, and memory cards. Retain files, deleted remnants, and filesystem metadata.
  • Tertiary Storage: Offline media like magnetic tapes and optical discs in jukeboxes. Used for long-term archives and backups.
  • Virtual & Cloud Storage: VM disks (VMDK/VHD) and object stores (AWS S3, Azure Blob). Persist across data centers with versioning and replication metadata.

Volatile vs Non-Volatile Memory

Understanding volatility is essential to capture data before it’s lost.

  • Volatile Memory: DRAM (RAM) and CPU caches (L1/L2/L3) that lose contents when power is removed. Use live-capture tools like Belkasoft RAM Capturer or FTK Imager RAM.
  • Non-Volatile Memory: Magnetic disks/tapes, NAND flash media, optical discs, and cloud object stores. Use bit-stream imaging, wear-leveling analysis, compression control, and API-driven exports.

Key Forensic Objectives by Medium

  • Magnetic Media: Preserve exact bit copies; watch for host-protected areas and proprietary sector layouts.
  • Flash Media: Account for TRIM operations and over-provisioned regions; capture entire NAND image when possible.
  • Optical Media: Disable hardware compression and log every disc swap or library action.
  • Cloud Object Stores: Script API pulls to capture data plus full metadata; reconcile eventual consistency and audit logs.

Storage Types and Forensic Implications

Categories define what you’re dealing with; types define how you access it. Selecting the wrong approach risks evidence gaps or contamination.

Direct Attached Storage (DAS)

  • Locally connected block devices (SATA, NVMe, USB).
  • Locally connected block devices (SATA, NVMe, USB).
  • Use hardware write-blockers and bit-stream tools (`dd`, FTK Imager, Guymager).
  • Beware BIOS/UEFI quirks that hide or remap NVMe partitions.
  • SSDs demand TRIM-aware capture and may require vendor SDKs for forensic-grade dumps.

Network Attached Storage (NAS)

  • File-level shares via NFS, SMB/CIFS, or AFP.
  • Leverage snapshots/exports to preserve ACLs, timestamps, and extended attributes.
  • If mounting risks writes, capture packets (`tcpdump`, Wireshark) and reconstruct transfers offline.
  • Document filesystem types (Btrfs, ZFS) and deduplication settings that affect deleted-data visibility.

Storage Area Network (SAN)

  • Block volumes over Fibre Channel or iSCSI.
  • Map LUNs and zoning with administrators before imaging.
  • Handle multipath carefully; use vendor utilities (EMC Navisphere, NetApp SnapManager).
  • SAN taps can intercept FC frames or iSCSI commands when direct admin access is limited.

Cloud Storage

  • RESTful APIs expose object and file stores.
  • Use SDKs/CLI (`aws s3api`, `az storage blob`) to download data plus metadata.
  • Track versioning, replication status, and correlate with audit logs.

Archival & Tape Libraries

  • Sequential media managed by robotic libraries or autoloaders.
  • Plan for extended seek times and multiple cartridge swaps.
  • Control compression settings; log each load/unload event for chain of custody.

Categories vs Types: Why the Distinction Matters

Categories answer “What kind of persistence and role does this storage have?” Types answer “How is this storage exposed to me?”

A secondary storage device might present as DAS, NAS, SAN, or a cloud-mounted block. Confusing category with type can lead to wrong acquisition methods—mounting when you should image or scripting APIs against a live filesystem—and may invalidate evidence in court.

Final Thoughts

  • Choose the right tools—from RAM grabbers to tape robotics or cloud SDKs.
  • Preserve both volatile execution artifacts and long-term archives.
  • Document a clear, defensible chain of custody and methodology.
